# VoltDesk: Frontend Structural & Layout Architecture

## The Global Layout Philosophy and Mobile-First Methodology
The structural foundation of the VoltDesk frontend is engineered with an uncompromising mobile-first methodology, meaning every container natively defaults to consuming one hundred percent of the viewport width before progressively enhancing into complex grid and flexbox arrangements on larger displays. The application is entirely encapsulated within a fixed, single-page viewport measuring exactly 100vw by 100vh, completely eliminating standard document scrolling in favor of internally managed overflow containers. The typography relies heavily on the JetBrains Sans font family to deliver a crisp, highly legible, and developer-centric aesthetic, perfectly suited for high-information density interfaces. The color palette operates on a foundation of neutral slate and zinc tones—acting as a quiet, low-contrast backdrop—punctured by high-visibility Sticker Mule orange (`#ea580c`) for primary actionable components and a distinct, soft purple spectrum reserved exclusively to highlight machine-generated AI drafts.

## The Customer Widget Structure
When a user first encounters the customer-facing interface, the structure is deliberately minimized to avoid obscuring the host application. On desktop environments, it manifests as a strictly bounded absolute container anchored to the bottom-right coordinate of the screen, consuming roughly twenty percent of the horizontal space (max-width of 350px) and a fixed height of about sixty percent of the vertical space (around 500px). However, on mobile viewports, this bounded box aggressively expands to claim one hundred percent of both the width and height, converting from a floating widget into a dedicated full-screen overlay. 

The internal architecture of this widget is a strict CSS flexbox column. The top ten percent is dedicated to a fixed-height header containing the brand title and a minimize button. The central eighty percent of the vertical space is allocated to the primary chat thread. This is a scrollable flex container utilizing a column-reverse directional flow, ensuring that new messages naturally anchor to the bottom boundary. Inside this thread, individual chat bubbles are restricted to a maximum width of eighty-five percent of the container, utilizing flex-end and flex-start alignments to visually separate customer inputs from agent responses. The final bottom ten percent houses the interaction zone: a horizontal flex row containing a rounded text input container spanning eighty percent of the row width, adjoined by a high-contrast orange submit button consuming the remaining twenty percent, engineered with optimal touch targets for mobile thumbs.

## The Agent Dashboard Architecture
The agent dashboard is architected to operate with the strict, zero-waste efficiency of a tiling window manager environment, ensuring every percentage of screen real estate is purposefully allocated without overlapping clutter. The root layout is a horizontal flex container spanning the entire screen. On a mobile device, this horizontal split collapses entirely; the interface defaults to showing a list view taking one hundred percent of the width, which then completely swaps to the chat view upon selection. On standard desktop and laptop displays, the layout rigidly divides the screen into two permanent master-stack partitions.

The left-hand sidebar, acting as the live queue, strictly consumes exactly twenty-five percent of the total horizontal viewport. This container is structurally divided into a static header taking the top ten percent for filtering controls (open versus resolved toggles), while the remaining ninety percent is a vertically scrolling flex list. Each individual queue item within this list is a padded rectangular flex block, utilizing a space-between layout to push the customer identification string to the left and the timestamp to the far right, ensuring rapid visual scanning. 

The remaining seventy-five percent of the horizontal screen acts as the primary operational stage: the active conversation window. This right-hand partition is another flex column divided into three distinct vertical zones. The uppermost five percent is a static contextual header displaying the current session status and a termination button. The middle seventy-five percent forms the deeply scrollable message history, structurally identical to the customer view but expanded across a wider horizontal plane to allow for comfortable reading of long-form technical support queries. 

## The AI Integration UI Components
The most critical structural element lies in the bottom twenty percent of the agent's active window, housing the input and AI interaction mechanics. Instead of a single static text box, this area is a stacked flex column containing the "Smart Draft" staging area positioned directly above the manual input bar. When the Go backend dispatches an AI-generated payload, a distinct container rendered in a soft purple background instantly materializes in this space. This staging box consumes ninety-five percent of the available width, featuring a JetBrains Sans text block for the suggested response, flanked by two explicitly distinct action buttons: a solid purple "Accept & Send" button and a subtle, ghosted "Discard" button. Below this dynamic staging area sits the permanent, manual text input row, stretching one hundred percent across the bottom margin, ensuring the agent can bypass the AI seamlessly without breaking their keyboard-first workflow.
